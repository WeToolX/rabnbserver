# 团队大小区接口开发文档

## 1. 需求背景

原用户团队列表接口为：

- `POST /api/user/team/list`
- 位置：`src/main/java/com/ra/rabnbserver/controller/user/UserController.java`
- 当前逻辑：按当前登录用户的 `path + id` 查询所有下级，支持 `targetLevel` 相对层级筛选，返回普通用户分页列表。

本次需求不是继续返回普通团队成员列表，而是按“当前用户的直属下级”拆分大区、小区，并返回每个区的矿机业绩。因此建议新增接口，不复用原接口，避免影响旧前端兼容。

## 2. 当前项目现状

### 2.1 技术栈

- Spring Boot
- MyBatis-Plus
- MySQL
- Sa-Token 登录鉴权
- 统一响应：`ApiResponse.success/error`

### 2.2 团队关系字段

用户实体：`src/main/java/com/ra/rabnbserver/pojo/User.java`

关键字段：

- `id`：用户 ID
- `user_wallet_address`：用户钱包地址
- `parent_id`：直属上级 ID
- `level`：树层级，根用户为 1
- `path`：祖先路径，格式类似 `0,1,2,`
- `direct_count`：直属下级人数
- `team_count`：全部下级人数
- `user_grade`：自动用户等级，从 1 开始
- `custom_user_grade`：后台人工等级，`User#getUserGrade()` 会返回自动等级和人工等级的较大值

本次大小区只统计直属下级，应使用：

```sql
WHERE u.parent_id = #{currentUserId}
```

不要用原接口的 `path LIKE currentUser.path + currentUser.id + ',%'`，否则会把孙级、更多层级也算进区入口。

### 2.3 矿机字段

矿机实体：`src/main/java/com/ra/rabnbserver/pojo/UserMiner.java`

关键字段：

- `user_id`：矿机所属用户 ID
- `wallet_address`：矿机所属用户地址
- `nft_burn_status`：卡牌销毁/兑换状态，`1` 表示兑换矿机成功
- `is_electricity_paid`：是否已经交过电费，`1` 表示已交电费
- `status`：矿机生命周期状态，`0` 待激活，`1` 运行中，`2` 已过期
- `create_time`：矿机记录创建时间，可作为“最后一台兑换矿机时间”
- `payment_date`：最近一次交电费时间

项目购买矿机流程是先落库 `user_miner`，再调用链上销毁卡牌，销毁成功后更新 `nft_burn_status = 1`。因此“已购/兑换矿机”建议只统计 `nft_burn_status = 1`，避免把链上销毁失败或异常重试中的矿机算入业绩。

## 3. 等级与电费分成现有逻辑

当前项目已经有等级与分成基础：

- 配置对象：`src/main/java/com/ra/rabnbserver/VO/MinerSettings.java`
- 配置键：`MINER_SYSTEM_SETTINGS`
- 等级开关字段：`activeMinerGradeMode`
- 阶梯配置：`tiers`，元素字段为 `grade`、`minCount`、`ratio`
- 等级重算入口：`MinerServe#recalculateUserGrade(Long userId)`
- 团队大小区预留重算入口：`MinerServe#recalculateUserGradeForTeamArea(Long userId)`

等级判断开关语义：

- `activeMinerGradeMode = true`：按直属下级已激活矿机数量判断等级，当前代码条件为 `user_miner.status = 1`
- `activeMinerGradeMode = false`：按直属下级已兑换矿机数量判断等级，当前代码条件为 `user_miner.nft_burn_status = 1`

电费分成比例：

- 根据当前用户最终等级 `User#getUserGrade()` 找到 `tiers` 中相同 `grade` 的 `ratio`
- 自动等级从 1 级开始，未达到任何阶梯时为 1，比例默认为 0
- 后台人工等级如果高于自动等级，会参与最终分成比例

新增大小区接口查询前建议调用：

```java
minerServe.recalculateUserGradeForTeamArea(userId);
```

这样前端打开大小区页面时，当前用户等级可以按最新直属下级矿机数量刷新。

## 4. 业务口径

### 4.1 业绩定义

每个直属下级代表一个区，该区业绩包含：

- 地址：直属下级 `user_wallet_address`
- 已购/兑换矿机数量：该直属下级自己的 `user_miner.nft_burn_status = 1` 数量
- 已激活矿机数量：该直属下级自己的已激活矿机数量
- 最后一台兑换矿机时间：该直属下级 `nft_burn_status = 1` 矿机中的最大 `create_time`

推荐“已激活”口径与等级开关保持一致使用 `status = 1`，因为需求写明“已经交电费激活”，当前项目交电费后会设置 `status = 1`。如果产品希望“曾交过电费但已过期”也算激活展示，可改为 `is_electricity_paid = 1`，但要和前端文案确认。

### 4.2 大区规则

在当前用户所有直属下级中：

1. 已兑换矿机数量最多的直属下级为大区。
2. 如果已兑换矿机数量相同，比较最后一台兑换矿机时间，时间更早者为大区。
3. 如果数量和时间都相同，建议按 `user_id ASC` 稳定排序。

排序表达：

```sql
ORDER BY
    purchased_count DESC,
    CASE WHEN last_exchange_time IS NULL THEN 1 ELSE 0 END ASC,
    last_exchange_time ASC,
    user_id ASC
```

没有兑换矿机时 `last_exchange_time` 为 `NULL`，应排在最后。

### 4.3 小区规则

除大区外的所有直属下级均为小区，按同一业绩规则从大到小排列：

```sql
ORDER BY purchased_count DESC, last_exchange_time ASC, user_id ASC
```

## 5. 新接口设计

### 5.1 接口

建议新增：

```http
POST /api/user/team/area-list
```

保留旧接口 `/api/user/team/list` 不动。

### 5.2 请求参数

最终接口按 `type` 返回大区或小区数据：

- `type = 1`：查询大区
- `type = 2`：查询小区

大小区判断必须基于全量直属下级，不能先分页再判断大区。接口可以支持分页，但分页只用于 `type = 2` 的小区列表；`type = 1` 固定返回大区单条数据。

建议 DTO：

```java
package com.ra.rabnbserver.dto.team;

import lombok.Data;

@Data
public class TeamAreaQueryDTO {
    /**
     * 1-大区，2-小区
     */
    private Integer type;

    private Integer page = 1;
    private Integer size = 20;
}
```

### 5.3 响应结构

接口统一返回列表结构，便于前端按同一个字段渲染：

- `type = 1`：`records` 中最多 1 条，即大区
- `type = 2`：`records` 中返回小区列表，已排除大区

```json
{
  "type": 2,
  "records": [
    {
      "userId": 15,
      "address": "0x...",
      "teamCount": 23,
      "purchasedCount": 6,
      "activeCount": 4,
      "lastExchangeTime": "2026-04-21 11:00:00"
    }
  ],
  "total": 3,
  "page": 1,
  "size": 20,
  "totalPurchasedCount": 16,
  "totalActiveCount": 11,
  "currentUserGrade": 2,
  "currentUserElectricityRatio": 0.15
}
```

建议新增 VO：

```java
package com.ra.rabnbserver.VO.team;

import com.fasterxml.jackson.annotation.JsonFormat;
import lombok.Data;

import java.math.BigDecimal;
import java.time.LocalDateTime;
import java.util.List;

@Data
public class TeamAreaItemVO {
    private Long userId;
    private String address;
    /**
     * 该直属下级自身团队总人数，取 user.team_count
     */
    private Integer teamCount;
    private Integer purchasedCount;
    private Integer activeCount;

    @JsonFormat(pattern = "yyyy-MM-dd HH:mm:ss")
    private LocalDateTime lastExchangeTime;
}

@Data
public class TeamAreaResultVO {
    /**
     * 1-大区，2-小区
     */
    private Integer type;
    private List<TeamAreaItemVO> records;
    private Long total;
    private Integer page;
    private Integer size;
    private Integer totalPurchasedCount;
    private Integer totalActiveCount;
    private Integer currentUserGrade;
    private BigDecimal currentUserElectricityRatio;
}
```

## 6. 实现方案

### 6.1 推荐落点

新增/修改文件：

- `src/main/java/com/ra/rabnbserver/dto/team/TeamAreaQueryDTO.java`
- `src/main/java/com/ra/rabnbserver/VO/team/TeamAreaItemVO.java`
- `src/main/java/com/ra/rabnbserver/VO/team/TeamAreaResultVO.java`
- `src/main/java/com/ra/rabnbserver/mapper/UserMapper.java`
- `src/main/java/com/ra/rabnbserver/server/user/UserServe.java`
- `src/main/java/com/ra/rabnbserver/server/user/impl/UserServeImpl.java`
- `src/main/java/com/ra/rabnbserver/controller/user/UserController.java`

`UserController` 只负责鉴权、获取当前用户 ID、返回响应；统计和组装放到 `UserServeImpl`。

`UserServeImpl` 需要注入 `MinerServe`，查询前调用等级重算入口。

### 6.2 Mapper SQL

建议在 `UserMapper` 中新增：

```java
@Select("""
    SELECT
        u.id AS userId,
        u.user_wallet_address AS address,
        u.team_count AS teamCount,
        COALESCE(COUNT(um.id), 0) AS purchasedCount,
        COALESCE(SUM(CASE WHEN um.status = 1 THEN 1 ELSE 0 END), 0) AS activeCount,
        MAX(um.create_time) AS lastExchangeTime
    FROM user u
    LEFT JOIN user_miner um
        ON um.user_id = u.id
       AND um.nft_burn_status = 1
    WHERE u.parent_id = #{userId}
    GROUP BY u.id, u.user_wallet_address, u.team_count
    ORDER BY
        purchasedCount DESC,
        CASE WHEN lastExchangeTime IS NULL THEN 1 ELSE 0 END ASC,
        lastExchangeTime ASC,
        u.id ASC
    """)
List<TeamAreaItemVO> selectDirectAreaStats(@Param("userId") Long userId);
```

如果最终确认“已激活”要统计曾经交过电费的矿机，把 `um.status = 1` 改为 `um.is_electricity_paid = 1`。

### 6.3 Service 逻辑

```java
@Override
public TeamAreaResultVO getTeamAreaList(Long userId, TeamAreaQueryDTO query) {
    if (query == null) {
        query = new TeamAreaQueryDTO();
    }
    if (query.getType() == null || (query.getType() != 1 && query.getType() != 2)) {
        throw new BusinessException("type 必须为 1-大区 或 2-小区");
    }

    minerServe.recalculateUserGradeForTeamArea(userId);

    List<TeamAreaItemVO> areas = baseMapper.selectDirectAreaStats(userId);
    TeamAreaResultVO result = new TeamAreaResultVO();
    result.setType(query.getType());
    result.setPage(query.getPage());
    result.setSize(query.getSize());

    if (areas == null || areas.isEmpty()) {
        result.setRecords(List.of());
        result.setTotal(0L);
        result.setTotalPurchasedCount(0);
        result.setTotalActiveCount(0);
        fillCurrentGradeAndRatio(result, userId);
        return result;
    }

    TeamAreaItemVO bigArea = areas.get(0);
    List<TeamAreaItemVO> smallAreas = new ArrayList<>(areas.subList(1, areas.size()));

    if (query.getType() == 1) {
        result.setRecords(List.of(bigArea));
        result.setTotal(1L);
    } else {
        result.setRecords(pageSmallAreas(smallAreas, query));
        result.setTotal((long) smallAreas.size());
    }
    result.setTotalPurchasedCount(sumPurchased(areas));
    result.setTotalActiveCount(sumActive(areas));
    fillCurrentGradeAndRatio(result, userId);
    return result;
}
```

`fillCurrentGradeAndRatio` 可以读取当前用户 `getUserGrade()`，再从 `MinerSettings.tiers` 按 `grade` 匹配 `ratio`。当前 `MinerServeImpl` 内已有类似私有方法 `findRatioByGrade`，如果不想重复逻辑，可以后续把等级结果查询抽成公共方法。

### 6.4 Controller 逻辑

```java
@SaCheckLogin
@PostMapping("/team/area-list")
public String getTeamAreaList(@RequestBody(required = false) TeamAreaQueryDTO query) {
    Long userId = getFormalUserId();
    TeamAreaResultVO result = userService.getTeamAreaList(userId, query);
    return ApiResponse.success("获取成功", result);
}
```

`UserServe` 增加方法：

```java
TeamAreaResultVO getTeamAreaList(Long userId, TeamAreaQueryDTO query);
```

## 7. 索引建议

为保证聚合查询性能，建议检查或新增：

```sql
ALTER TABLE user ADD INDEX idx_user_parent_id (parent_id);
ALTER TABLE user_miner ADD INDEX idx_user_miner_user_burn_status_time (user_id, nft_burn_status, status, create_time);
```

如果最终激活口径使用 `is_electricity_paid`：

```sql
ALTER TABLE user_miner ADD INDEX idx_user_miner_user_burn_paid_time (user_id, nft_burn_status, is_electricity_paid, create_time);
```

如果表中已有同等前缀索引，不要重复添加。

## 8. 测试用例

### 8.1 无直属下级

当前用户没有 `parent_id = currentUserId` 的用户。

期望：

- `bigArea = null`
- `smallAreas = []`
- `totalPurchasedCount = 0`
- `totalActiveCount = 0`

### 8.2 只有一个直属下级

期望：

- 该直属下级为大区
- 小区为空

### 8.3 多个直属下级，兑换数量不同

数据：

- A 已兑换 10，已激活 6
- B 已兑换 5，已激活 5
- C 已兑换 0，已激活 0

期望：

- A 为大区
- 小区顺序：B、C

### 8.4 兑换数量相同，比较最后兑换时间

数据：

- A 已兑换 10，最后兑换时间 `2026-04-20 10:00:00`
- B 已兑换 10，最后兑换时间 `2026-04-19 10:00:00`

期望：

- B 为大区，因为最后一台矿机兑换时间更早

### 8.5 存在待销毁或异常矿机

数据：

- A 有 5 条 `user_miner`
- 其中 3 条 `nft_burn_status = 1`，2 条 `nft_burn_status = 0`

期望：

- A 已兑换统计为 3

### 8.6 已激活统计

数据：

- A 有 5 条 `nft_burn_status = 1`
- 其中 4 条 `status = 1`

期望：

- A 已兑换 5
- A 已激活 4

### 8.7 等级开关

配置：

- `activeMinerGradeMode = true`

期望：

- 当前用户等级按直属下级 `status = 1` 的矿机数量重算

配置：

- `activeMinerGradeMode = false`

期望：

- 当前用户等级按直属下级 `nft_burn_status = 1` 的矿机数量重算

## 9. 注意事项

1. 新接口保留 `@SaCheckLogin`，继续用 `getFormalUserId()` 获取正式用户 ID。
2. 大区/小区必须在全量直属下级范围内判断，不能先分页再取第一页第一条当大区。
3. 原 `/api/user/team/list` 不建议修改，避免影响旧团队列表。
4. `lastExchangeTime` 主要用于排序，即使前端不展示，也建议返回，方便排查排序结果。
5. `已购/兑换矿机` 建议严格使用 `nft_burn_status = 1`。
6. `已激活` 推荐使用 `status = 1`；如果产品定义为“曾交过电费”，再切换为 `is_electricity_paid = 1`。
7. 当前项目源码注释存在部分编码显示异常，新增文件建议统一 UTF-8。
